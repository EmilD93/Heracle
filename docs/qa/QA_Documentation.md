### 1.2 Цел и обхват

**Цел:** Да се дефинират тестовите дейности, подходи и ресурси, необходими за осигуряване на качеството на продукта. Този Test Plan покрива функционалното тестване на Authentication и Events модулите.

**Обхват:**
- Вход в системата (Login)
- Регистрация на потребители (Register)
- Управление на събития (Create, Read, Update, Delete)
- Валидация на форми и полета
- Сигурност и оторизация

**Извън обхвата:**
- Performance тестване
- Security penetration testing (основен скрийнинг се прави)

### 1.3 Тестова стратегия

| Аспект | Подход |
|--------|--------|
| **Тип тестване** | Функционално, Regression, Smoke |
| **Методология** | Black-box testing |
| **Нива на тестване** | Integration, System |
| **Приоритизация** | P1 (Critical), P2 (High), P3 (Medium), P4 (Low) |

### 1.4 Тестова среда

| Компонент | Детайли |
|-----------|---------|
| Браузъри | Chrome (latest), Firefox (latest), Edge (latest) |
| Устройства | Desktop (Windows 10/11), Mobile responsive |
| Тестови среди | Staging: `https://staging.[project].com` |
| Инструменти | Postman (API), DevTools, BrowserStack (при нужда) |

### 1.5 Рискове и митигация

| Риск | Вероятност | Въздействие | Митигация |
|------|------------|-------------|-----------|
| Закъснение на dev builds | Средна | Високо | Ранна комуникация с dev екипа, приоритизиране на smoke тестове |
| Непълна документация | Висока | Средно | Срещи с Product Owner за изясняване на изискванията |
| Промени в изискванията | Средна | Високо | Гъвкав подход, редовно обновяване на тестовете |

### 1.6 Критерии за изход

- 100% от P1 тестовете преминават
- Максимум 5 отворени P2 бъга
- Няма критични (P1) бъгове в отворено състояние
- Code coverage ≥ 80% за покритите модули

### Дефиниции за Severity

| Ниво | Описание | Примери |
|------|----------|---------|
| **Blocker** | Системата е неизползваема | Не може да се влезе в системата; Crash при стартиране |
| **Critical** | Основна функционалност не работи | Регистрацията не изпраща имейл; Events не се записват в БД |
| **Major** | Функционалност работи, но със сериозни проблеми | Валидацията пропуска невалидни имейли; Бавно зареждане (>10s) |
| **Minor** | Козметичен проблем или лесно заобикалящ се | Неправилно подравняване; Грешно съобщение за грешка |
| **Trivial** | Незначителен проблем | Typo в етикет; Липсващ hover ефект |

## 4. Бележки от обучения

По време на подготовката за този проект преминах през следните материали:

### Software Testing Fundamentals (GeeksforGeeks)
- **Линк:** https://www.geeksforgeeks.org/software-testing/software-testing-tutorial/
- **Ключови концепции:** SDLC & STLC, Levels of Testing (Unit, Integration, System, Acceptance), Types of Testing (Functional, Non-functional, Regression, Smoke)
- **Приложение в проекта:** Използвах знанията за дефиниране на тестовите нива и типове в Test Plan-а. Реших да започна с Smoke тестове за Authentication, тъй като без Login останалите функционалности са недостъпни.

### How to Write Test Cases (BrowserStack)
- **Линк:** https://www.browserstack.com/guide/how-to-write-test-cases
- **Ключови концепции:** Good test case structure (Preconditions, Steps, Expected Result), Traceability to requirements, Independence of test cases
- **Приложение в проекта:** Приложих структурата Precondition → Steps → Expected Result във всички тестови сценарии. Всяка стъпка е атомарна и независима. Добавих Traceability чрез ID-та (AUTH-###, EVT-###).

### Bug Reporting Best Practices (Dev.to)
- **Линк:** https://dev.to/hamidsaymon/overview-of-the-test-case-component-bug-report-in-software-testing-2284
- **Ключови концепции:** Clear reproduction steps, One bug per report, Environment details, Severity vs Priority distinction
- **Приложение в проекта:** Създадох структуриран Bug Report Template с ясно разделение на стъпки, очакван/актуален резултат и среда. Дефинирах Severity нива с конкретни примери от проекта за по-лесна класификация.
